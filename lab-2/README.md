В рамках лабораторной работы вам предлагается реализовать одну из предложенных
классических структур данных (список, дерево, бинарное дерево, hashmap,
граф...)

Текущий вариант: `rb-tree-lazy` (красно-чёрное дерево с ленивыми вычислениями)

Требования:

1. Функции:
    - добавление и удаление элементов
    - фильтрация
    - отображение (map)
    - свертки (левая и правая)
    - структура должна быть моноидом
2. Структуры данных должны быть неизменяемыми
3. Библиотека должна быть протестирована в рамках unit testing
4. Библиотека должна быть протестирована в рамках property-based тестирования
(как минимум 3 свойства, включая свойства моноида).
5. Структура должна быть полиморфной
6. Требуется использовать идиоматичный для технологии стиль программирования
7. Обратите внимание:
    - API должно быть реализовано для заданного интерфейса и оно не должно
    "протекать". На уровне тестов -- в первую очередь нужно протестировать
    именно API (dict, set, bag)
    - Должна быть эффективная реализация функции сравнения (не наивное
    приведение к спискам, их сортировка с последующим сравнением),
    реализованная на уровне API, а не внутреннего представления

## Ключевые элементы реализации

Структура представляющая узел дерева:

```rust
type NodeRef<K, V> = Rc<RefCell<Node<K, V>>>;
type WeakNodeRef<K, V> = Weak<RefCell<Node<K, V>>>;

struct Node<K, V>
where
    K: Ord,
{
    pub key: K,
    pub value: V,
    pub color: Color,
    pub left: Option<NodeRef<K, V>>,
    pub right: Option<NodeRef<K, V>>,
    pub parent: Option<WeakNodeRef<K, V>>,
}
```

Для преставления ссылок на дочерние элементы используется обёртка с
использованием `Option<Rc<RefCell<...>>>`, для родительских:
`Option<Weak<RefCell<...>>>`

Для возможности возвращения из узлов дерева ссылок на значение элементов
была создана специальная структура `Memory<T>`, представляющая собой
*абстрактный аллокатор* для элементов с типом `T`, при вставке и удалении
в дерево сначала производится вызов к данной структуре, после чего
элементу присваивается определённый идентификатор, по которому можно
получить непосредственно сам элемент

Само дерево представлено следующей структурой:

```rust
struct Tree<K, V>
where
    K: Ord,
{
    root: Option<Rc<RefCell<Node<K, usize>>>>,
    memory: Memory<V>,
}
```

> Как можно заметить во внутреннем представлении используются узлы вида
> `Node<K, usize>`, параметризированный тип `V` в данной реализации
> выносится в поле `Memory<V>`

Свойства моноида, обеспечиваются за счёт нейтрального элемента (пустого дерева)
и операции слияния (`Tree::merge`)

Для реализации ленивых операций над данными дерева (`map`, `fold`, `filter`)
используются итераторы, т.е. дерево приводится к итератору при помощи метода
`into_iter` и вспомогательной структуры `TreeIterator<'a, K, V>`:

```rust
struct TreeIterator<'a, K, V>
where
    K: Ord + Clone,
    V: Clone,
{
    stack: Vec<Rc<RefCell<Node<K, usize>>>>,
    current: Option<Rc<RefCell<Node<K, usize>>>>,
    tree: &'a Tree<K, V>,
}
```

## Реализованные тесты

### Unit-тесты

#### Вставка элементов в дерево

**Сигнатура**: `tree::tests::test_insert`

**Цель**: Проверка корректности вставки и получения элементов из дерева

**Методика**: Добавление нескольких элементов в дерево и последующее
получение этих элементов

#### Удаление элементов из дерева

**Сигнатура**: `tree::tests::test_delete`

**Цель**: Проверка корректности вставки и получения элементов из дерева

**Методика**: Добавление нескольких элементов в дерево, удаление
некоторых из вставленных элементов и последующее получение этих элементов

#### Слияние нескольких деревьев

**Сигнатура**: `tree::tests::test_merge`

**Цель**: Проверка корректности слиния двух деревьев

**Методика**: Наполнение двух деревьев различными данными, их слияние и
последующий анализ полученного дерева

#### Проверка работы итератора

**Сигнатура**: `iterator::tests::test_iter`

**Цель**: Проверка работоспособности итератора, полученного из дерева

**Методика**: Наполнение дерева данными, получение итератора, проверка
ожидаемого и действительного ключа и значения на каждой итерации

#### Проверка корректности операций с данными

**Сигнатуры**: `iterator::tests::test_map`,
`iterator::tests::test_filter`, `iterator::tests::test_map`

**Цель**: Проверка правильности значений, полученных при выполнении
операций с итератором

**Методика**: Наполнение дерева данными, получение итератора, выполнение
операций `map`, `fold` и `filter` и проверка ожидаемых результатов на
соответствие действительным

#### Проверка выполнения ленивых вычислений

**Сигнатуры**: `iterator::tests::test_lazy`

**Цель**: Проверка того, что расчёты производятся при обращении к данным,
а не при вызове функции обработки

**Методика**: Наполнение дерева данными, применение операции, которая
выбрасывает исключение при попытке обработки некоторых данных, получение
элемента, с которым исключение не выбрасывается

#### Проверка операции сравнения

**Сигнатуры**: `tree::tests::test_partial_eq`,
`tree::tests::test_partial_eq_empty_trees`,
`tree::tests::test_partial_eq_different_structures`

**Цель**: Проверка реализации операции реализации операции сравнения из
трейта `PartialEq`

**Методика**: Наполнение деревьев различными, попарное сравнение и проверка
результатов сравнения

### Properties-based тесты

#### Проверка на соблюдение правил построения КЧ-дерева

**Сигнатуры**: `tree::tests::test_black_root_property`,
`test_black_nodes_property`

**Цель**: Проверка построенного библиотекой дерева на соответствие
[принципам организации](https://ru.wikipedia.org/wiki/%D0%9A%D1%80%D0%B0%D1%81%D0%BD%D0%BE-%D1%87%D1%91%D1%80%D0%BD%D0%BE%D0%B5_%D0%B4%D0%B5%D1%80%D0%B5%D0%B2%D0%BE#%D0%9F%D1%80%D0%B8%D0%BD%D1%86%D0%B8%D0%BF_%D0%BE%D1%80%D0%B3%D0%B0%D0%BD%D0%B8%D0%B7%D0%B0%D1%86%D0%B8%D0%B8)
красно-чёрных деревьев

**Методика**: Создание различных деревьев с различной структурой и их
дальнейшей проверкой

#### Проверка на свойства моноида

**Сигнатура**: `tree::tests::test_monadic_properties`

**Цель**: Проверка свойств моноида у реализованного дерева

**Методика**: Создание нескольких деревьев, сложение их с нейтральным
элементом, проверка на эквивалентность исходного дерева и суммы (т.е. того что
$A \cup \emptyset = A$)

#### Проверка глубины дерева

**Сигнатура**: `tree::tests::test_depth`

**Цель**: Проверка на удовлетворение логарифмического роста дерева в
зависимости от количества элементов

**Методика**: Создание нескольких деревьев с $2^n$ элементов, расчёт глубины
каждого дерева, расчёт коэффициента $r^2$ для выборки глубин дерева и
линейно растущей последовательности
